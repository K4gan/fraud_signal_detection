# Fraud signal detection

Imbalanced binary classification project for card-not-present and mixed-channel transaction screening. The project avoids external data dependencies by generating a controlled transaction stream with fraud signals, noise and realistic class imbalance.

## Project scope

- Synthetic transaction data with amount, velocity, distance, device age and channel features.
- Three sklearn models: logistic regression, random forest and histogram gradient boosting.
- Threshold selection from the precision-recall curve instead of a fixed `0.50` cutoff.
- CLI scoring for one transaction at a time.
- Notebook for visual checks around class imbalance and risk features.

## Dataset

Generated by `make_transaction_data()` in `train.py`.

Target: `is_fraud`.

Important features:

- `amount`: transaction amount, log-normal distribution.
- `merchant_risk`: merchant-level risk score.
- `velocity_1h`, `velocity_24h`: short window transaction velocity.
- `distance_from_home_km`: rough behavioral distance proxy.
- `device_age_days`: newer devices are higher risk.
- `payment_channel`: card-present, ecommerce, wallet or bank transfer.
- `country_match`: whether customer and transaction country match.

## Run

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python train.py
```

Example scoring call:

```bash
python predict.py \
  --model boosting \
  --amount 289.90 \
  --hour 2 \
  --merchant-risk 0.46 \
  --customer-age-days 42 \
  --velocity-1h 5 \
  --velocity-24h 13 \
  --distance-from-home-km 96 \
  --device-age-days 3 \
  --payment-channel ecommerce \
  --country-match no
```

## Practical note

Fraud detection is rarely a pure accuracy problem. The training script stores a per-model operating threshold chosen on the holdout set using the precision-recall curve, then `predict.py` reuses that threshold.
